## きっかけログ（要約）

```text
user: 大きい気があるね          ← 音声: おそらく「大きい木があるね」
topics: polar_bear:大きい, sniffer:大きい
stance: hypothesis
allowed: シロクマ, スニッファー  enforce_wl=True
LLM: …sensed… → unusable_output
fallback/骨子: 俺には見えんけど、シロクマかスニッファーあたりかもしれんな
```

場所は森。シロクマもスニッファーも文脈としておかしい。  
**ASR ミスだけでなく、SM が「大きい」だけで identify 経路に引きずっている。**

再現（照合）:

| 入力 | topic ヒット |
|---|---|
| 大きい気があるね | polar_bear / sniffer（`大きい`） |
| 大きい木があるね | 同上（木でも同じ） |
| 木があるね | なし |
| きれいな木 | glow_squid / tropical_fish（`きれい`） |

---

## 問題の層（4段）

### 1. 音声認識（入力のゆらぎ）

- 「木」→「気」などは避けられない。  
- **ASR を完全修正するより、SM が弱い手がかりで断定しない**方が重要。  
- 誤変換があっても「木の話」に戻せる余地を LLM 雑談側に残す。

### 2. トピック照合が広すぎる（検索面）

- poetic `visual_tags` の **形容・汎用語**（大きい・きれい・白い…）が全文検索キーになっている。  
- 1 語ヒットだけで score が通る（`大きい` 長さ3 × weight → 18 など）。  
- **旗・ババアのような識別力のある語**と、**景色の形容**が同じエンジンで同等扱い。

### 3. stance が topic あり＝即 hypothesis（経路）

```text
if topic_hits: return hypothesis
```

結果:

- ヒント載せる  
- 白リスト enforce オン  
- S3 骨子「〜かもしれんな」  
- プロンプトも identify 用 policy  

**「大きい木」雑談が identify 専用レールに乗る。**  
会話が成立しない主因はここ。

### 4. 場所・観測と無関係な候補（文脈）

- 森なのにシロクマ（雪・極地）を同列候補にしている。  
- 観測ゼロでも、弱いタグだけで候補確定。  
- structure の biome 突合（F′）は structure 用。**mob 候補には場所フィルタが無い。**

副次: LLM が英語混入（`sensed`）→ unusable → 誤った骨子が表面化する。

---

## 設計原則（合意したいこと）

1. **弱い手がかりでは identify しない。** 雑談を壊さない。  
2. **識別力のある語**（旗・ババア・とんがり帽子・前哨基地）は今どおり効かせる。  
3. **観測（visual / passive / hearing）があるときの種名は正当。**  
4. **場所と明らかに矛盾する候補は落とす or 大幅減点。**（完璧なスポーン表は不要、できる範囲で）  
5. ASR は「誤りうる」前提。誤っても **none/clarify に落とせば LLM が雑談できる。**

## 対策（レイヤ別）

### T1. 汎用語タグの「単独では弱くする」（照合）

**やること:**

- **単独では hypothesis に使わない語**の集合（GENERIC_TOPIC_TERMS）  
  例: 大きい、小さい、きれい、白い、黒い、赤い、青い、長い、丸い、速い、怖い、変な…  
- ヒットが **GENERIC のみ**のとき:
  - score を大幅減 or **候補から除外**
  - または hit は残すが **identify ゲートを通さない**（推奨は後者＋前者の併用）

**やらないこと:**

- 汎用語を JSON から全削除（描写・川柳用には残す価値あり）  
- 「旗」「ババア」など識別語を GENERIC に入れない  

**実装案:**

```text
find_catalog_topics の後処理 or スコア時:
  matched がすべて GENERIC → その entry の score を 0 扱いで落とす
  または matched に非 GENERIC が1つも無い hit を捨てる
```

「大きい＋雪」なら polar_bear は残しうる。「大きい」だけなら落とす。

### T2. hypothesis の入場条件を厳しくする（stance）— **最優先**

**現状:** topic 1 件でも hypothesis。  
**変更案:**

```text
saw:     視認 / ついさっき視認（変更なし）
hypothesis: topic があり、かつ次のどれか
  A. identify 意図（何/誰/あいつ/あれ何 等）
  B. 高信頼マッチ（非 GENERIC 語、または score 閾値＋ top が独走）
  C. 観測ブースト付き（visual/passive と topic が一致）
none:    上記以外（弱い topic は hints も載せない or 載せるが stance は none）
clarify: 弱い「あれ？」のみ
```

**重要:**  
「大きい木があるね」は identify 意図が薄い → **topic が残っていても none**。  
LLM は雑談。白リスト enforce もオフ（現行 none どおり）。

ヒントの扱い:

| 案 | 内容 |
|---|---|
| **推奨** | stance≠hypothesis なら **catalog_topic_hints を details に載せない**（引きずり防止） |
| 弱い案 | hints は載せるが policy は none（モデルがヒントに引っ張られやすい） |

S3 骨子:

- **hypothesis かつ高信頼のときだけ**（現行＋「非 GENERIC 必須」を追加）  
- 同点2頭の「シロクマかスニッファー」は **弱い GENERIC 由来なら禁止**

### T3. 場所・バイオームとの整合（スコア）

完璧なスポーン DB は後回しでよい。段階:

**3a. scene_tags と粗い場所の一致（すぐできる）**

- 候補 mob の `poetic.scene_tags`（雪・沼・海・洞窟…）  
- 現在 biome group / place_context のキーワード  
- 不一致なら score 減点 or 非表示  
- 例: scene「雪」＋ 森 → polar_bear 減点  

**3b. habitat_groups（任意・弱い用途のみ）**

- 粗い `aquatic` / `snowy` / `nether` 等は付けてもよい  
- **用途は「観測ゼロの弱いテキスト仮説」の減点だけ**  
- **観測（visual / passive / hearing）がある種は habitat を無視**（動物園・連れ込み・展示）  
- 全 mob の厳密 `spawn_biomes` 一括整備は不要（メンテ重・動物園と矛盾しやすい）

森でシロクマが並ぶのは **T-A/B + 3a だけでもかなり減る**。habitat は補助。

### T4. unusable / fallback の表面症状

- `sensed` 英語 → unusable は正しい。  
- 問題は **fallback が誤った identify 骨子**になること。  
- T2 で hypothesis に入らなければ、骨子も出ず中立/LLM 雑談になる。  

追加:

- unusable 時、**弱い hypothesis 骨子より** 中立 fallback や none 向け短文を優先。

### T5. ASR（後回し・補助）

- 主要誤変換辞書（気↔木 等）は **任意・薄い**。  
- 依存しすぎない。T1+T2 で十分なケースが多い。  
- 必要なら normalize に数語だけ。

## 将来: VLM（プレイヤー巨大建造物・景色）

**方針: 入れるつもり。カタログでは無理な「見た感想」の本命。**

### なぜ必要か

| 構造化（今）で足りる | カタログで足りない |
|---|---|
| バニラ mob / 公式 structure / 音 | **プレイヤーが建てた巨大建造物** |
| 旗・ウィッチ・前哨の知識 | 例: **巨大なヒカキンの像** |
| biome・structure 生成可否 | 形のユーモア・スケール・「誰に似てるか」 |

巨大像には entity id も structure id も無い。  
相棒が一緒に見て感心・ツッコミするには **画（VLM）か同等の視覚要約**が要る。

### 役割分担（確定イメージ）

```text
構造化（今〜強化）
  mob / 音 / 公式 structure / 近くの block id
  → 正確な種名・戦闘・前哨ありうる

VLM（将来・イベント駆動）
  「あれ見て」「すごいだろ」「これ作った」
  巨大建造物・像・変な形・でかい木の景色
  → 感想・スケール・雰囲気
  → 厳密 id は自信がなければ曖昧に（像っぽい／人型のでっかいの 等）
```

- **動物園・連れ込み** → 構造化観測優先（habitat で否定しない）  
- **ヒカキン像クラス** → VLM の本戦場  
- **でかい木** → T-A/B で雑談は今でも可能。VLM があれば「正面のでっかい木」まで上がる加点  

### 起動条件（案・常時スキャンしない）

1. プレイヤー意図: あれ見て／これ何／すごいだろ／感想求む 等  
2. または「巨大っぽい」ヒューリスティック（高さ・占有）＋任意  
3. **先に構造化**（視線付近の visual / passive）  
   - ヒットあり → それで答える（VLM 不要）  
   - ヒットなし or 景色・建造物っぽい → VLM  
4. VLM も曖昧 → clarify（聞き返し）

### 出力・衝突ルール

- 感想寄り。カタログ白リストで **落とさない**（カタログ外が正解）  
- 近くに脅威がいれば **戦闘／脅威優先**、景色モードでは偽モブ identify 骨子を出さない  
- レイテンシ・コストのため **低頻度・トリガー限定**

### 今やっておくと楽な準備（実装は後で可）

| 準備 | 内容 |
|---|---|
| stance 枠 | 将来 `scenery` / `build_comment` を空けておく意識（T-A 設計と両立） |
| event schema | スクショ or 視点画像の **任意フィールド**を後から足せるようにする |
| 当面 | **T-A〜T-E** で弱いタグ過適合を止め、VLM 待ちでも木の雑談が死なないようにする |

### やらないこと（VLM 周り）

- mob 分類の本命を VLM に置き換える（entity 観測の方が正確）  
- 毎フレーム VLM  
- VLM 完了まで T-A/B を先送りする
