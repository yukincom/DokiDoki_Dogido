# 技術課題

## 1. Minecraft 側からどうデータを取るか

- 周辺モンスター情報
- 明るさ
- インベントリ
- 周辺資源
- 時刻
- プレイヤー死亡要因

これらをリアルタイムに取得する必要がある。

### 候補

- Mineflayer
- Fabric / Forge Mod

### 現時点の判断

- `mindcraft` は参考実装として活用する
- ただし、今回の主対象は bot 自身ではなく人間プレイヤー本人
- そのため、Minecraft Java 版では Fabric クライアント mod を第一候補とする
- Mineflayer は取得できる情報の参考と比較対象として扱う

## 2. ステート管理が複雑

扱いたい状態が多い。

- 怖がりレベル
- うるさいと言われた回数
- 直前のモンスター記憶
- 暗さ対処フロー

これを LLM だけに任せるとブレやすい。

### 方針

- ステートはコードで管理する
- LLM には状態とイベントを注入して発話だけを担わせる

## 3. 「うるさい」検知

以下のような入力をまとめて解釈したい。

- うるさい
- 静かにして
- 黙れ

これは小さな NLU 問題になる。

### 方針

- 最初はルールベースで始める
- 必要なら同義語辞書を増やす
- 早い段階で誤検知率を確認する

## 4. 重要な設計判断

この企画の面白さは、LLM の自由応答よりも優先制御にある。

そのため、先に固めるべきなのは以下。

- イベントの種類
- イベントの優先順位
- 発話抑制ルール
- 状態遷移
- 音から得た情報をどこまで断定してよいか
- 緊急音声と通常 TTS の分離

## 5. 実装の優先順位（2026-07-09 更新）

### 骨格（概ね完了）

1. イベントスキーマ定義
2. ステートマシン定義
3. 優先度ルール定義
4. サンプルイベントでの発話テスト

### いま磨く

- 対話品質（プレイヤー入力・状況理解・発話抑制のバランス）
- 暗所は専用イベントより snapshot スコア + 多段状態（経緯は [現行仕様 §6](current-spec.md)）

### 対話設計が固まってから

- M5Stack 出力
- 定時お知らせ・LINE / Discord などの外部メッセージ連携（`yuno-chan-api` 流用可否の再検討を含む）

### 使わない方針

- Hermes などの汎用エージェント基盤（機能過剰）
- 将来の LLM ワークフロー整理は LangChain / LangGraph を想定
